# GTFS notes: 2024 prototype and the current 2026 feed

This note keeps the useful ideas from the original notebook close to the current data work. It is not a standards audit and it does not choose a routing model.

## Data used for the comparison

The workbench downloads the official Rapid KL rail feed from:

`https://api.data.gov.my/gtfs-static/prasarana?category=rapid-rail-kl`

A live download was parsed and inspected on 14 July 2026. That snapshot had SHA-256:

`079a1b5d72cbecd923a7271468c45572d36b60d2947acb497f440d1bd4888467`

It contained:

| Table | Rows |
| --- | ---: |
| `agency.txt` | 1 |
| `calendar.txt` | 8 |
| `frequencies.txt` | 106 |
| `routes.txt` | 8 |
| `shapes.txt` | 7,280 |
| `stop_times.txt` | 1,122 |
| `stops.txt` | 187 |
| `trips.txt` | 48 |

The eight routes were Ampang, Kelana Jaya, Sri Petaling, Kajang, Putrajaya, KL Monorail, BRT Sunway, and the Shah Alam Line. The feed produced 16 distinct route-direction stop patterns.

A public snapshot committed on 6 July 2026 in `NightfuryEquinn/Ridership` at commit `b9a97003ae95346f9e7fbafe81a1e6a685d08253` had only seven routes, 167 stops, and 42 trips; it did not yet contain the Shah Alam Line. That short-lived difference is a useful reminder that the official endpoint is mutable. Experiments should record the acquisition time and checksum rather than refer to a generic “2026 feed.”

The official ZIP also contained hidden macOS AppleDouble files under `__MACOSX/`. They had `.txt` names but binary contents. The loader ignores this packaging debris and records only real GTFS tables in provenance.

## What the 2024 notebook learned

| 2024 approach | Useful insight | Risk or limitation | Current treatment |
| --- | --- | --- | --- |
| Loaded the main GTFS tables plus frequencies and shapes into pandas | DataFrames are convenient for exploration and plotting | The notebook created a large amount of global state | `GTFSFeed` keeps raw tables together and returns small derived DataFrames |
| Read a `route_id` directly from `stops.txt` | It made route filtering quick | `stops.route_id` is not standard GTFS and can be misleading | Route membership is derived through `stop_times.trip_id -> trips.route_id` |
| Patched Kajang Line station membership manually | The notebook correctly noticed that the feed's helper column was unreliable | The patch depended on station-code patterns | No patch is needed when normal GTFS joins are used |
| Inferred line names and day types from `trip_id` text | The identifiers were human-readable and convenient | Identifier formatting is not a stable data contract | Use `trips.route_id`, `trips.service_id`, and `trips.direction_id`; retain `trip_id` as an opaque string |
| Parsed service times as datetimes and used `.dt.seconds` | It enabled quick duration calculations | It wraps or discards day information and cannot represent extended service-day hours safely | GTFS times are integer seconds from service-day start |
| Built adjacent-stop travel times from ordered stop records | This is the core raw material for later graph experiments | Aggregation and service-period choices were not yet explored | Parsing exposes ordered stop patterns without selecting an aggregation method |
| Estimated waiting from `frequencies.txt` | Frequency windows are useful inputs for later experiments | A single waiting formula should not be baked into parsing | Frequency rows and parsed time bounds are exposed for later use |
| Added explicit walking interchange edges | The feed did not fully describe physical transfers | Proximity alone produced false positives | Curated interchange dispositions remain separate machine-readable evidence |
| Used shapes, colours, and coordinates to plot the network | Visual feedback made the work easier to understand | Plotting was mixed into notebook state | `plot_network()` provides a reusable first plotting layer |
| Tried Peartree and sketched a custom `GTFSGraph` | Existing tools revealed useful ideas and limitations | Graph design was started before the feed representation was settled | Graph and routing choices remain outside the parsing workbench |

## Important observations in the live feed

### Normal GTFS relationships are more reliable than helper columns

The current `stops.txt` includes non-standard columns such as `category`, `route_id`, `geometry`, `isOKU`, `status`, and `search`. These may be helpful for inspection, so the parser preserves them.

They should not be authoritative. Kajang Line stops use a helper route value that differs from the actual `trips.route_id=KGL`. The old manual Kajang correction is therefore unnecessary when station membership is derived from trips and stop times.

### `stop_times.txt` also has non-standard route data

The feed adds `route_id` and `direction_id` to `stop_times.txt`. In 690 stop-time rows, the helper route value differs from the corresponding `trips.route_id`; examples include `AGL -> AG`, `KJL -> KJ`, `SPL -> PH`, and `SAL -> SA`.

The workbench keeps the raw value as `stop_times_route_id` in joined views and uses `trips.route_id` as the normal GTFS relationship.

### Identifier strings remain descriptive, but should stay opaque

Trip IDs still look like `KGL_MonFri_0`, so the 2024 string-splitting approach would often appear to work. The same information is already present in explicit `route_id`, `service_id`, and `direction_id` columns. Parsing the identifier adds fragility without adding information.

### Service-day time parsing is necessary

The current feed uses unpadded values such as `6:00:00` and has 18 frequency windows ending at `24:00:00`. The parser accepts arbitrary non-negative hour values, including later values such as `27:30:45`, and does not wrap at midnight.

### Frequency data remains useful

The feed has route- and service-specific frequency windows. This retains one of the useful inputs used by the 2024 notebook, while leaving future waiting-time interpretation open.

### Transfer evidence still needs another source

The live feed has no `transfers.txt` and no `parent_station` column. The existing curated interchange evidence therefore remains relevant.

All ten stop IDs referenced by the current curated interchange records resolved in the live feed: `AG2`, `AG8`, `KG15`, `KG17`, `KJ15`, `MR1`, `MR3`, `PY17`, `SP2`, and `SP8`.

### The quality report is descriptive, not punitive

The live feed produced informational findings for non-standard columns, missing optional transfer/calendar-exception tables, extended frequency times, helper-route disagreements, and absent station hierarchy. It produced no broken-reference, invalid-coordinate, malformed-time, or negative/non-monotonic-time errors.

## What this work deliberately does not decide

- how trips should be aggregated into graph edges;
- whether routing should be timetable-aware;
- how waiting should be estimated;
- how transfers should be penalized;
- how meeting candidates should be generated;
- what makes one meeting point better than another.

The parser's job is to make those experiments quick to start, easy to inspect, and grounded in a specific recorded feed snapshot.
